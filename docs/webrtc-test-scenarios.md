# webrtc-test-scenarios

## 目的
このファイルは、webrtc3 の修正後に最低限確認したいシナリオをまとめたチェックリストである。
毎回 full test を行うためではなく、再発しやすい経路を漏らさず確認するために使う。

---

## 基本ルール
- A 端末と B 端末の 2 台で確認する
- 可能なら PC とスマホの組み合わせでも確認する
- 「接続できた」だけでなく、「相互に見える」「相互に聞こえる」を分けて確認する
- 「subscribe できた」と「UI に表示された」を分けて確認する
- UI 修正時は、PC とスマホで表示優先順位が意図どおりかも確認する

---

## シナリオ 1: 既存参加者と新規参加者の相互表示

### 手順
1. A が先に room へ入室する
2. A の camera / mic publish を確認する
3. B が同じ room へあとから入室する

### 期待結果
- A から B の映像・音声が見える
- B から A の映像・音声が見える
- B だけ既存参加者を見られない状態にならない
- 自分自身の remote tile が出ない

### 主に確認したいこと
- `subscribeExisting()` が late joiner 側で効いているか
- `onStreamPublished` 側と責務が競合していないか
- attach 先 DOM が未準備のまま終わっていないか

---

## シナリオ 2: 先に入った人が新しく入った人を見られる

### 手順
1. A が先に入室する
2. B があとから入室する

### 期待結果
- A から B の映像・音声が見える
- 新規配信通知の経路だけは正常かどうかを分けて判断できる

### 主に確認したいこと
- `onStreamPublished` 系の購読経路
- self publication 除外条件
- join 前後の race condition

---

## シナリオ 3: leave 後の再入室

### 手順
1. A と B が同じ room に入る
2. A が leave する
3. A が同じ room に再入室する

### 期待結果
- A が再入室できる
- A と B が再び相互に見える
- 前回の remote tile や publication 状態が壊れたまま残らない

### 主に確認したいこと
- leave 時の cleanup
- event handler の解除漏れ
- 再 join 後の購読再初期化

---

## シナリオ 4: 自分の publication を自分で受けない

### 手順
1. A が単独で入室する
2. local preview を確認する
3. remote tile 側も確認する

### 期待結果
- local preview は表示される
- 自分の remote tile は出ない
- self publication を subscribe していない

### 主に確認したいこと
- publisher.id と local member.id の比較
- self 除外条件の効き方

---

## シナリオ 5: 画面共有開始

### 手順
1. A と B が入室する
2. A が画面共有を開始する

### 期待結果
- B に A の共有画面が見える
- A 側の local preview と UI state が壊れない
- 既存の camera publication と新しい stream 参照が競合しない

### 主に確認したいこと
- share 開始時の publication 差し替え
- attach の更新
- mute / UI state の引き継ぎ

---

## シナリオ 6: 画面共有停止後に camera へ戻る

### 手順
1. A が画面共有中の状態を作る
2. A が画面共有を停止する

### 期待結果
- A の映像が camera に戻る
- B に A の camera が見える
- A 側 local preview も camera に戻る

### 主に確認したいこと
- camera stream 再生成
- publication 参照更新
- local preview attach 更新

---

## シナリオ 7: 背景ぼかし ON / OFF

### 手順
1. A と B が入室する
2. A が背景ぼかしを ON にする
3. A が背景ぼかしを OFF にする

### 期待結果
- ON 時に B から A の映像が見える
- OFF 後に通常 camera へ戻る
- local preview が壊れない
- 何度か切り替えても破綻しない

### 主に確認したいこと
- processor 初期化と dispose
- processed stream と normal stream の差し替え順
- publication と attach の整合

---

## シナリオ 8: ミュート状態の維持

### 手順
1. A が入室する
2. A が audio mute / video mute を切り替える
3. 必要なら stream 差し替え操作を行う
4. B から状態を確認する

### 期待結果
- mute / unmute が反映される
- publish 差し替え後も必要な state が維持される
- UI 表示と実際の publish 状態が食い違わない

---

## シナリオ 9: 話者ハイライト

### 手順
1. A と B が入室する
2. A が話す
3. B が話す

### 期待結果
- 話している参加者に対応する tile へハイライトが付く
- 別参加者へ正しく切り替わる
- tile 更新や stream attach 修正の後でも消えていない

### 主に確認したいこと
- memberId と tile の対応
- attach 後の container 管理
- ハイライト付与 / 除去処理

---

## シナリオ 10: room 指定

### 手順
1. URL クエリに room を付けて入室する
2. 別端末でも同じ room で入室する

### 期待結果
- 同じ room に入る
- room 指定で join / late join が壊れない

### 主に確認したいこと
- room 決定ロジック
- URL クエリ優先の整合

---

## シナリオ 11: モバイルで主要映像が初見で見える

### 手順
1. スマホで room に入室する
2. 共有なし状態の画面を確認する
3. 共有あり状態の画面も確認する

### 期待結果
- スマホで横スクロールしなくても主要映像へ到達できる
- 「見えていないだけ」を接続不具合と誤認しにくい
- 共有ありでは共有画面が先に見える

### 主に確認したいこと
- モバイル時の主表示位置
- 横スクロール前提になっていないか
- 共有あり / なし で優先順位が破綻していないか

---

## シナリオ 12: モバイルで参加者カメラが自然に見える

### 手順
1. PC とスマホで同じ room に入る
2. スマホ側から参加者カメラの見え方を確認する
3. 共有あり / なし 両方で確認する

### 期待結果
- 参加者カメラが縦積みまたはグリッドで自然に読める
- 横フィルムストリップ前提になっていない
- 共有画面と参加者カメラの見た目責務が混ざっていない

### 主に確認したいこと
- スマホ時の参加者レイアウト
- 補助表示の扱い
- 共有サムネイルと参加者カメラが競合していないか

---

## 保留シナリオ: room 固定化を実装したら追加で確認すること

### 手順
1. URL クエリなしで A が入室する
2. URL クエリなしで B が入室する
3. 再読み込み、leave、再入室も試す

### 期待結果
- 毎回 URL を変えなくても同じ room に入れる
- room 固定化で既存の join / leave / late join が壊れない
- 意図しない room 競合が起きない

---

## 実施メモ欄
- 再現条件が端末依存か
- PC → スマホ、スマホ → PC で差があるか
- 受信失敗なのか、表示失敗なのか
- コンソールエラーや warning の有無
- スマホでは主要映像が初見で見えるか
