# AGENTS.md

## 目的
このリポジトリは Vue SFC、JavaScript の composable / service、必要に応じて Python 補助スクリプトを含む WebRTC プロジェクトである。
既定の編集方針は保守的にする。
要求を満たす最小の正しい変更を優先し、依頼されていない範囲へ拡張しない。

## 常設ルール
- 変更対象は必要なファイルだけに限定する
- 既存挙動は、依頼で明示された場合を除き維持する
- 無関係なリファクタはしない
- コメント、コメントアウト、文言、整形、空白、改行コード、引用符スタイルは依頼がない限り触らない
- 既存の公開 API、export、関数名、変数名、ファイル名、コンポーネント名は必要な場合を除き変えない
- 大きな差分より局所的な差分を優先する
- 未コミット変更がある前提で、巻き戻ししやすい差分にする
- merge marker、重複パッチ、文字化けを絶対に残さない
- 実行していない検証を成功したとは書かない

## スコープ判断
編集前に必ず次を整理する。
- 変更対象ファイル
- 必要な挙動変更
- 触ってはいけない範囲
- 現状から目的までの最小経路

1 回の依頼が 1〜3 個の局所編集で終わるなら、その形を優先する。
読みやすさ改善だけを目的に広く書き換えない。

## リポジトリ前提
- フロントエンドは Vue + JavaScript を前提とする
- WebRTC の主要責務は composable と service に分かれている前提で扱う
- SkyWay 周りは既存の接続フローを尊重し、必要な箇所だけ直す
- UI 文言はユーザー要求の言語に合わせる
- 日本語コメントは完全保持する

## `.vue` 編集ルール
- `<template>`, `<script>`, `<style>` の並びを変えない
- API スタイルの変換はしない
- 小さな UI 変更のためにテンプレート全体を組み替えない
- 既存の class 構造やレイアウトは、依頼がない限り維持する
- 表示テキストは編集箇所だけ変更する
- 日本語 UI を求められた場合、編集箇所の表示文言は日本語に統一する
- 無関係な文言の翻訳はしない

## フロントエンド `.js` 編集ルール
- module shape と export style を維持する
- composable / service の API shape は必要な場合を除き変えない
- モジュール再設計より、小さな helper 追加を優先する
- 無関係な関数移動はしない
- 非同期フローは、バグ箇所でない限り組み替えない
- module 間で受け渡す object shape を不用意に変えない

## `.py` 編集ルール
- 関数名、CLI 引数、ファイル構成は必要な場合を除き維持する
- 無関係な整形をしない
- 明示依頼がない限り依存追加をしない
- 構造変更より、ロジックや path の局所修正を優先する

## WebRTC プロジェクト専用ルール
- `useStreamReceiver.js` は orchestrator として扱い、順序制御と UI state を優先する
- DOM 直操作を増やしすぎない。DOM 寄りの処理は既存 service へ寄せる判断を優先する
- late join, subscribeExisting, onStreamPublished, join / leave 再入室の整合を重視する
- 画面共有、ミュート、背景ぼかし、RNNoise は既存フローを壊さない
- 画面共有や背景ぼかしの修正では、publish / unpublish / 再 attach の順序破壊を避ける
- 見た目修正では、状態管理や接続処理まで広げない
- モバイル UI では、主要映像や主要操作を横スクロール前提にしない

## コメントと文字コード
- 既存コメントは保持する
- コメントの翻訳、要約、書き換えはしない
- コメントアウトは依頼がない限り触らない
- 日本語テキストを含むファイルでは文字化け防止を最優先する

## 許可する変更
- 小さなロジック修正
- state の受け渡し修正
- ライフサイクルの局所修正
- イベント処理の局所修正
- 必要最小限の UI 文言修正
- 変更箇所に限定した小さな helper 追加
- 変更箇所に限定した検証

## 禁止事項
- 大規模リファクタ
- whole-file rewrite
- 無関係な import 並び替え
- 無関係な整形
- コメント変更
- 安定 API の rename
- 不要な build config / lockfile / package 変更
- 無関係な画面や module の同時変更
- 動いている実装の別アーキテクチャ置換

## 差分品質
良い差分の条件:
- 必要ファイルだけ変更している
- 必要行だけ変更している
- 近傍のコードスタイルを維持している
- formatting-only 変更がない
- conflict marker がない
- 文字化けがない

悪い差分の兆候:
- 依頼に比べて差分が大きすぎる
- コメントまで変わっている
- import や関数位置が大量に動いている
- 1 行機能のために関数全体や template 全体を書き換えている

## 検証ルール
可能な範囲で最小の有用な検証を行う。
優先順:
1. 変更ファイルの syntax check
2. 変更箇所に近い lint / typecheck
3. 関連する軽量 build / test
4. 必要な場合のみ full build

検証できない場合は、実行できなかった内容を明記する。

## Skill 適用ルール
- 実行時挙動に影響する変更では `code-change-verification` Skill を適用してから作業する
- WebRTC の接続・購読・late join・画面共有・背景ぼかし・RNNoise の不具合修正では `webrtc-bugfix` Skill を適用してから作業する
- 見た目や表示文言の最小修正では `ui-minimal-fix` Skill を適用してから作業する
- `readable-code` Skill が明示された場合は、変更箇所に限ってコメント追加・改善を許可する

## 完了報告ルール
完了時は必ず次を簡潔に書く。
- 変更ファイル
- 依頼された挙動変更
- 未検証部分
- 制約や残課題

## 曖昧な依頼
曖昧さがあっても、まず最も狭い妥当解釈を採用する。
安全な最小修正が不可能な場合だけ確認を挟む。
